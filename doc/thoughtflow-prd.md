# ThoughtFlow (思流) 产品需求文档 (PRD)

## 1. 产品定位
**ThoughtFlow** 是一款基于 AI 的轻量化知识整理应用。它能够以多轮采集会话承接日常零散想法和参考网址，在信息一致的前提下先发散补全、再收敛成可归档知识内容，并通过 AI 语义分析、分类、专题刷新和检索回溯，逐步形成动态增长的个人知识体系。

*   **核心价值**：从“随意记录”到“会话整理”再到“体系产出”的自动化流水线。
*   **部署目标**：极致轻量，单文件部署，本地优先。

---

## 2. 核心技术架构 (Architecture)
*   **开发语言**：Golang (高性能并行处理，单二进制文件)。
*   **基础应用框架**：使用 [`magicCommon/framework`](https://github.com/muidea/magicCommon) 作为应用生命周期与运行单元框架。
    *   统一承载模块注册、`Setup` / `Run` / `Teardown` 生命周期、EventHub 事件协同、BackgroundRoutine 后台任务、配置管理与监控能力。
    *   入口层显式选择启用的运行单元，避免业务包通过隐式 import 触发模块注册。
*   **网络服务框架**：使用 [`magicEngine`](https://github.com/muidea/magicEngine) 作为 HTTP 与实时通信框架。
    *   用于实现 REST API、路由注册、中间件、静态资源服务、文件上传能力。
    *   预留 SSE 能力，用于异步加工进度、索引状态、Git 同步状态等实时推送。
*   **持久化层**：
    *   **存储格式**：纯 Markdown 文件 (Markdown-first)，人类可读。
    *   **版本控制**：集成 Git，实现全自动的变更追踪与多端同步。
*   **索引/分析层**：
    *   **数据库**：DuckDB (嵌入式 OLAP 数据库)。
    *   **检索技术**：混合搜索 (关键词 FTS + 向量 Embedding)。
*   **AI 引擎**：接入大模型 API (DeepSeek/OpenAI 兼容格式)。

### 2.1 后端运行单元划分
基于 `magicCommon/framework` 将后端拆分为可独立装配的运行单元，保障轻量部署下的清晰边界：

*   **capture**：负责文本/URL 多轮采集会话、会话草稿持久化、归档为原子 Markdown、采集事件发布。
*   **refiner**：负责网页正文清洗、AI 摘要、标签生成、Embedding 生成等后台加工任务。
*   **topic**：负责专题规则、专题命中判断、专题主文档智能缝合与手动重组。
*   **search**：负责 DuckDB 索引、FTS/向量混合检索、检索结果聚合。
*   **git-sync**：负责自动 `git add / commit`、变更记录、后续多端同步扩展。
*   **application**：负责将上述能力暴露为 `magicEngine` HTTP API 和静态 UI 入口，不直接持有其他运行单元的正式状态。

运行单元之间通过 EventHub 或明确的窄接口协作；正式业务状态由所属运行单元维护，跨单元写入和状态变更必须走事件命令或应用服务编排。

### 2.2 API 与服务边界
`magicEngine` 负责对外服务层，`magicCommon/framework` 负责应用内部生命周期和模块协同：

*   **HTTP API**：
    *   `POST /api/thoughts`：提交文本或 URL。
    *   `GET /api/thoughts/{id}`：查看原子笔记与元数据。
    *   `POST /api/capture/sessions` / `GET /api/capture/sessions/active`：创建或恢复采集会话。
    *   `POST /api/capture/sessions/{id}/messages`：向采集会话追加对话消息并刷新上下文。
    *   `POST /api/capture/sessions/{id}/archive`：通过菜单或对话意图将会话归档为 Thought。
    *   `POST /api/thoughts/{id}/reopen-session`：基于已归档 Thought 发起补充完善会话。
    *   `GET /api/search`：执行关键词、语义或混合检索。
    *   `POST /api/topics` / `PUT /api/topics/{id}`：创建与维护专题规则。
    *   `POST /api/topics/{id}/rebuild`：手动触发专题重组。
    *   `POST /api/topics/{id}/refresh`：基于 Thought 和采集会话信息刷新专题候选与正式视图。
*   **实时接口**：
    *   `GET /api/events`：基于 SSE 推送采集处理进度、索引状态、专题更新结果与 Git 提交结果。
*   **服务层约束**：
    *   `magicEngine` handler 只处理请求解析、鉴权、响应格式化和错误映射。
    *   业务流程进入对应运行单元的 service/biz 层，不在 handler 中直接操作 Markdown、DuckDB 或 Git。
    *   长耗时任务必须提交给 `magicCommon` BackgroundRoutine，HTTP 请求只返回任务受理状态或当前结果快照。

## 3. 功能需求 (Functional Requirements)

### 3.1 会话式采集 (The Capture)
*   **默认会话优先**：用户输入文本、URL 或问题时，默认只进入当前采集会话，不立即创建 `thoughts/` Markdown。
*   **会话延续**：未明确触发“新建会话”时，系统始终激活最后一个未归档会话；服务重启后也必须恢复该会话及其上下文。
*   **文本想法**：支持随手记下的短句、段落，并以多轮对话持续补充。
*   **URL 采集**：
    *   系统自动使用 `Colly/Jina Reader` 爬取正文。
    *   AI 自动提炼网页摘要、核心观点及元数据。
*   **上下文提取**：每轮对话后自动维护结构化上下文，包括主题、目标、已确认事实、冲突点、待澄清问题、候选标题、标签、摘要、关键点和归档草稿。
*   **发散与收敛**：多轮对话应在信息一致时先发散补充角度、例子、关联内容和问题，再逐步收敛为可归档内容；发现冲突时先澄清或标记冲突，不直接混入归档稿。
*   **归档触发**：只有用户明确要求保存、归档、提交或通过功能菜单触发保存时，才生成带 YAML 元数据的独立 Markdown 文件并存入 `thoughts/` 目录。
*   **对话式保存**：保存动作既可以由 UI 菜单触发，也可以由 LLM/本地意图识别从对话中触发，例如“保存这条”“归档成 thought”“把上面整理成笔记”。
*   **归档预览**：无论由对话还是菜单触发保存，都先展示归档预览，不直接写入 Thought；预览需包含标题、正文、标签、来源链接、关联专题和保存策略。
*   **归档策略**：普通会话默认创建新 Thought；从已归档 Thought 重新发起的会话默认生成“补充/修订 Thought”，并保留原 Thought backlink；更新原 Thought 或另存新 Thought 作为显式可选策略。
*   **安全更新**：更新原 Thought 前必须展示 diff 或内容变更预览，并由用户确认。
*   **会话上下文结构**：`session_context` 至少包含 `topic`、`goal`、`confirmed_facts`、`open_questions`、`conflicts`、`candidate_title`、`candidate_tags`、`candidate_summary`、`candidate_body`、`source_links`、`related_thought_ids`、`suggested_topic_ids`、`archive_intent`、`archive_strategy`。
*   **可编辑字段**：LLM 或本地规则自动维护上下文，用户可编辑候选标题、标签、正文和保存策略。
*   **收敛节奏**：每轮对话后自动刷新结构化上下文和候选归档稿；默认继续发散，直到用户触发整理、保存或归档。若系统判断信息足够，可提示是否归档，但不得自动保存。
*   **会话生命周期**：允许多个未归档会话；默认打开最近活跃的未归档会话。已归档会话保留历史但不再默认恢复；首版不自动清理长期未活跃会话。

### 3.1.1 Thought 重新对齐会话
*   **从 Thought 发起会话**：用户可基于任一已归档 Thought 创建新的补充会话。
*   **初始上下文**：新会话需加载原 Thought 的标题、原始内容、摘要、关键点、标签、关联专题、来源链接和 Git 记录摘要。
*   **非破坏性编辑**：重新对齐会话默认不直接修改原 Thought，直到用户明确选择保存策略。
*   **补充闭环**：会话中的新增信息应可被整理为修订稿、补充稿或新 Thought，并保留与原 Thought 的回溯关系。

### 3.2 AI 智能加工 (The Refiner)
*   **自动标签**：AI 识别内容并打上多维度标签。
*   **向量化 (Embedding)**：生成语义向量，存入 DuckDB，为关联分析做准备。
*   **异步处理**：利用 `magicCommon` BackgroundRoutine 执行后台非阻塞加工，确保录入体验秒级响应。
*   **进度事件**：加工状态通过 EventHub 发布，并由 `magicEngine` SSE 接口推送给前端。

### 3.3 多专题管理与自动进化 (Topic System)
*   **专题创建**：用户可定义多个专题（如：AI 研究、烹饪心得、投资笔记）。
*   **匹配规则**：每个专题支持设置关键词触发和语义相似度阈值。
*   **全量信息刷新**：专题创建、规则修改或用户触发刷新后，系统应基于全部相关信息自动刷新专题视图，包括已归档 Thought、未归档采集会话、会话结构化上下文、从 Thought 重新发起的补充会话和 synthesis 草稿。
*   **状态分层**：
    *   已归档 Thought 可正式进入专题成员关系和专题主文档。
    *   未归档会话只能作为候选线索、近命中建议或待确认内容，不默认写入专题主文档。
    *   从旧 Thought 发起的补充会话需要提示其可能影响原 Thought 或相关专题。
*   **自动路由与更新**：
    *   当新碎片命中专题规则时，AI 自动将其整合进专题主文档 (`topics/{name}/index.md`)。
    *   **智能缝合**：AI 负责寻找合适的章节插入内容，而非简单的末尾追加。
    *   专题刷新结果需区分“正式收录”“候选建议”“冲突/待确认”。
    *   专题创建、规则更新、会话上下文变化和 Thought 归档后自动刷新候选；专题主文档只基于正式 Thought 或用户确认后的内容更新。
*   **管理界面**：支持专题的创建、规则修改、大纲维护及手动重组触发。

### 3.4 即席检索与展示 (Ad-hoc Retrieval)
*   **混合搜索**：支持 `Ctrl+K` 唤起搜索框。
    *   **精确匹配**：基于 DuckDB 的 FTS 索引进行关键词搜索。
    *   **语义探索**：基于向量相似度查找“意思相近”的想法。
*   **即时合稿 (Synthesis-on-the-fly)**：
    *   支持在检索结果中手动勾选多个碎片。
    *   AI 即时生成针对选中内容的总结报告或逻辑大纲。
*   **溯源展示**：所有搜索结果和专题内容均可反向链接到最原始的原子笔记。

### 3.5 自动化 Git 工作流 (Git Integration)
*   **自动提交**：每次录入、专题更新后，系统自动执行 `git add / commit`。
*   **透明化记录**：Git Commit 记录作为知识进化的“心路历程”。
*   **任务化执行**：Git 操作作为 `git-sync` 运行单元的后台任务执行，失败时通过事件通知 UI，不阻塞采集主流程。

## 4. 界面需求 (UI Requirements)

### 4.0 Web 交互刷新原则
*   **保持主体框架**：保留现有工作台式产品形态、主导航菜单和页面级布局，不回退为营销页或大幅改动信息架构。
*   **页面内交互优化**：针对采集、笔记、搜索、专题、整理、设置等页面重新梳理主任务路径，减少跳转、重复入口和低价值说明。
*   **内容优先呈现**：优先展示用户正在整理的会话、候选归档稿、原子笔记正文、专题文档和搜索结果；运行状态、指标、调试信息默认折叠或移入次级区域。
*   **渐进披露**：复杂配置、任务详情、事件日志、模型参数、索引和 Git 诊断信息默认不占据主视图，只在用户需要时通过抽屉、详情区或设置页展开。
*   **一致操作模型**：保存、归档、新建会话、从 Thought 重新整理、加入专题、生成合稿等关键动作需在对应页面保持一致的位置、命名和反馈方式。
*   **简化冗余内容**：移除重复的状态卡、过长说明、开发调试字段和不影响决策的辅助文案，保留必要错误、风险和外部请求提示。
*   **单页主任务**：每个页面只突出一个主任务，降低并列操作数量；次要任务通过抽屉、折叠区或详情入口承载。

### 4.1 专题管理大盘 (Topic Dashboard)
*   展示所有专题卡片、统计数据（碎片数、字数）及最近活跃度。
*   提供专题创建引导视图。
*   展示专题刷新状态：正式成员、候选会话、近命中、冲突和待确认项。
*   专题列表应突出名称、描述、正式成员数、候选数和最近更新时间；详细规则和调试事件放入详情页或抽屉。

### 4.2 智能检索中心 (Search Hub)
*   响应式搜索界面，支持实时过滤。
*   搜索结果侧边栏预览功能。
*   搜索结果应突出标题、摘要、命中片段、标签、专题归属和可回跳来源；score、explain 和调试字段默认折叠。
*   支持从结果直接加入整理篮、打开 Thought、发起重新整理会话或生成专题候选。

### 4.3 专题详情工作台 (Topic Workspace)
*   **左侧**：专题体系化文档预览（Markdown 渲染）。
*   **右侧**：配置面板（规则设置、大纲编辑）。
*   **候选区**：展示未归档会话和补充会话对当前专题的影响建议，用户确认后才进入正式内容。
*   专题详情默认以文档和候选影响为中心；规则、大纲、提案、活动记录通过 tab 或抽屉进入，避免常驻占用阅读空间。

### 4.4 采集会话工作台
*   默认打开最后一个未归档会话，提供“新建会话”“归档为 Thought”“从 Thought 重新整理”入口。
*   显示多轮对话、结构化上下文、候选归档稿和冲突/待澄清问题。
*   支持通过菜单保存，也支持识别对话中的保存意图后让用户确认归档。
*   主视图应聚焦当前会话和候选归档内容；历史会话、上下文详情、模型推理反馈和任务事件放入侧栏或抽屉。
*   归档前需清晰展示即将保存的标题、正文、标签、来源链接和保存策略。
*   页面主任务是“继续整理当前会话并归档为 Thought”。

### 4.5 笔记与整理页面
*   Thought 详情应优先展示可阅读内容、AI 摘要、关键点、标签、来源和关联专题；Git commit、Job、索引状态等运行信息默认折叠。
*   整理/合稿页面应突出来源篮、草稿编辑器、保存策略和 source links；历史草稿和模板作为次级内容。
*   设置页面应以必要配置和状态修复为主，高级指标、事件流、诊断详情默认收起。
*   笔记页面主任务是“阅读 Thought 并发起重新整理”；搜索页面主任务是“查找、预览并加入整理”；专题页面主任务是“阅读正式文档并确认候选”；整理页面主任务是“基于来源篮生成并保存草稿”；设置页面主任务是“配置和健康修复”。

## 5. 非功能性需求 (Non-Functional Requirements)

*   **轻量化部署**：
    *   无需安装大型数据库中间件。
    *   单二进制执行文件，内存占用在空闲时应低于 50MB。
*   **数据安全与隐私**：
    *   数据存储在本地 Git 仓库，用户拥有绝对所有权。
    *   仅用户明确启用或功能必要的 LLM、Embedding、网页抓取/Reader 请求发送至云端 API；UI 需标识会触发外部请求的动作。
    *   未归档会话和结构化上下文属于本地草稿状态，不应进入 Git，除非用户明确归档。
    *   系统默认本地可用；外部 LLM、Embedding 和 Reader 能力通过配置启用，并在设置中全局声明。
    *   触发网页 Reader、LLM 整理或 Embedding 时，UI 使用轻量提示说明可能发生外部请求，不做每次阻塞确认。
*   **扩展性**：
    *   生成的 Markdown 目录结构需完美兼容 Obsidian 和 Logseq。
    *   新增采集源、LLM/Embedding Provider、检索策略和同步后端时，优先以 `magicCommon/framework` 运行单元或 focused package 扩展，避免堆叠到单一服务。
*   **可维护性**：
    *   后端入口保持单二进制启动，框架模块由入口显式装配。
    *   配置按全局配置与模块配置拆分，LLM/Embedding API key、DuckDB 路径、Git 策略、HTTP 端口均通过 `application.toml` 定义。
    *   关键任务需要暴露结构化日志和基础监控指标，便于定位采集失败、provider 调用失败、索引延迟和 Git 提交失败。

## 6. 核心业务流程 (Data Flow)

1.  **输入**：用户输入 `URL/文本/对话指令` -> `magicEngine` HTTP API 接收请求。
2.  **会话暂存**：`application` 调用 `capture` 会话能力 -> 追加消息、保存草稿、刷新结构化上下文；默认不生成 Thought。
3.  **发散整理**：LLM 或本地规则基于会话上下文追问、补全、提取候选标题/标签/摘要/关键点和待澄清问题。
4.  **收敛归档**：用户通过对话意图或菜单明确保存 -> `capture` 将会话归档为原子 Markdown -> 发布采集事件。
5.  **加工**：`refiner` 订阅事件或接收后台任务 -> 爬虫抓取 -> AI 摘要/向量化 -> 更新 `thoughts/` 元数据。
6.  **路由**：`search` 更新 DuckDB 索引 -> `topic` 查询匹配规则 -> 命中的专题触发 LLM 缝合任务 -> 更新 `topics/` 目录。
7.  **专题刷新**：专题规则创建、更新、归档事件或手动刷新触发后，topic 结合正式 Thought 与会话候选刷新正式成员和候选建议。
8.  **持久化**：`git-sync` 接收 Thought/Topic 变更事件 -> 执行 Git 自动提交 -> 发布提交结果事件。
9.  **消费**：用户通过 UI 搜索、采集会话、专题工作台或原子笔记详情回溯内容；前端通过 SSE 获取后台任务进度。

## 7. 遗留项清单 (Backlog)

1.  **会话恢复**：实现跨服务重启恢复最后一个未归档采集会话，并确保未显式新建会话时默认复用该会话。
2.  **会话上下文模型**：定义并持久化结构化上下文字段，覆盖主题、事实、冲突、待澄清问题、候选归档稿、保存意图和归档策略。
3.  **对话式保存识别**：补充 LLM/本地规则识别“保存/归档/提交”意图，并在必要时二次确认。
4.  **发散-收敛策略**：实现多轮对话中的自动追问、关联补充、冲突检测和归档稿收敛。
5.  **菜单保存入口**：在采集会话 UI 中补齐明确的“归档为 Thought”“新建会话”“从 Thought 重新整理”操作。
6.  **Thought 重新发起会话**：支持从已归档 Thought 加载上下文，生成补充会话，并提供更新原 Thought、另存新 Thought、生成补充 Thought 的保存策略。
7.  **专题全量刷新**：专题创建/更新后基于 Thought、未归档会话、补充会话和草稿刷新正式成员与候选建议，并区分候选状态。
8.  **候选内容确认流**：未归档会话命中专题时只进入候选区，需用户确认或归档后才能写入专题主文档。
9.  **Web 交互刷新**：在保留主体导航和页面布局的前提下，重做采集、笔记、搜索、专题、整理、设置页面的信息层级、主操作路径和内容呈现。
10. **页面简化**：移除或折叠重复状态、调试字段、冗长说明和低价值辅助内容，保证主视图聚焦知识内容与当前任务。
11. **隐私提示**：在 UI 和配置中明确标识 LLM 分类、Embedding、网页抓取和 Reader 回退等外部请求。
12. **验收测试**：补充会话恢复、对话保存、菜单保存、Thought 重新整理、专题候选刷新、Web 交互刷新和隐私提示的端到端测试。

## 8. 验收标准补充

1.  **会话恢复**：服务重启后打开采集页，自动恢复最近活跃的未归档会话及其 `session_context`。
2.  **归档预览**：用户通过菜单或对话表达保存意图后，系统展示归档预览，确认前不创建 Thought。
3.  **普通归档**：普通会话确认归档后创建新 Thought，并触发 refine、index、topic 和 git-sync 后续流程。
4.  **重新整理**：从已归档 Thought 发起会话后，默认保存为补充/修订 Thought，并保留原 Thought backlink。
5.  **更新保护**：选择更新原 Thought 时，必须展示 diff 或内容变更预览，确认后才写入。
6.  **上下文收敛**：连续多轮对话后，`session_context` 能持续更新候选标题、标签、摘要、正文、冲突和待澄清问题。
7.  **专题候选**：未归档会话命中专题时只出现在候选区，不写入专题主文档；归档或确认后才进入正式成员。
8.  **Web 收口**：采集、笔记、搜索、专题、整理、设置页面均保留主体导航和页面布局，主视图只突出各自单一主任务。
9.  **隐私提示**：触发 LLM、Embedding 或 Reader 外部请求时，UI 能给出轻量提示，设置页能查看全局外部请求配置。
